The Encryption Dilemma

01. The Legacy Approach

Traditional encryption architectures force developers to manage cryptographic keys within their own application state. This creates massive liability. If a database is breached, the keys are often found nearby.

  • Key rotation requires system downtime.
  • PII is exposed during database migrations.
  • Compliance boundaries (GDPR/DPDP) are hard to map.

02. Encryption-as-a-Service

Cipherion decouples data storage from data protection. By utilizing a proxy middleware or direct REST/gRPC APIs, fields are encrypted before they hit your database.

  • Zero-Key Architecture: Passphrases never touch our servers.
  • Dynamic Tokenization for analytics on masked data.
  • Field-level granularity down to the JSON key.

Developer Experience

Integrate field-level encryption in under 10 lines of code.

auth-service.ts 
import { Cipherion } from '@cipherion/node-sdk';

// Initialize the client with your scoped environment token
const cipherion = new Cipherion({
  apiKey: process.env.CIPHERION_API_KEY,
  enclaveRegion: 'ap-south-1'
});

async function registerUser(userData) {
  // Define which fields represent PII/Sensitive data
  const policy = {
    encrypt: ['ssn', 'credit_card'],
    tokenize: ['email'], // Preserves format for DB lookups
    ignore: ['created_at', 'tier']
  };

  // Data is encrypted locally or via secure proxy before DB insertion
  const securedPayload = await cipherion.protect(userData, policy);
  
  await db.users.insert(securedPayload);
  return { status: 200, message: 'User safely stored.' };
}

Hardware-Enforced Security(Nitro)

Software security is only as strong as the underlying OS. Cipherion leverages AWS Nitro Enclaves to create isolated compute environments to process your cryptographic materials. These enclaves have no persistent storage, no interactive access, and no external networking.

Even if Cipherion's parent EC2 instances are compromised, the memory containing the cryptographic operations and plaintext data inside the Nitro Enclave cannot be read by any user, not even a root user or an AWS administrator. Cryptographic attestation ensures the enclave is running the exact, untampered code.

Frequently Asked Questions

What exactly is "Zero-Key Architecture"?

In a standard setup, you store an encryption key in your environment variables. If your server is compromised, the key is compromised. With Cipherion's Zero-Key architecture, the master keys are derived per-session and exist only in the volatile memory of an isolated hardware enclave. Once the operation is complete, the memory is scrubbed.

How does this affect database performance?

Direct REST API calls add roughly 15-25ms of latency depending on region. However, if you deploy our Dockerized Proxy Middleware directly in your VPC, or use our gRPC endpoints, latency is typically under 3ms. By only encrypting specific fields (like PII), the overall query performance of your database remains largely unaffected.

Does Cipherion store any of my data?

No. Cipherion is strictly a compute layer. We receive plaintext, process it in memory, and return ciphertext (or vice versa). We do not have a persistent database that stores your payloads. We only store metadata related to your API usage for billing and audit logging purposes.

What happens if Cipherion goes down?

We operate across multiple availability zones within AWS. In the Enterprise tier, we deploy dedicated proxy instances directly into your infrastructure, meaning even if our control plane experiences an outage, your local encryption/decryption processes will continue uninterrupted using cached, short-lived derived keys.
